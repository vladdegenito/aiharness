## 1. SAST Findings

### Real Findings (in src/ / container/ / public/)

#### SAST-001 — Dockerfile runs as root (ERROR)
- **Rule:** `dockerfile.security.missing-user.missing-user`
- **File:** `container/Dockerfile:5`
- **Severity:** ERROR
- **Message:** No `USER` directive is present; the Python `server.py` process therefore runs as `root` inside the container. If the Semgrep container process is ever compromised (e.g., via a maliciously crafted scan payload that escapes Python's `tempfile` sandbox), the attacker would have root within the container.
- **Status:** Real finding. Not in test fixtures. Low exploitability in practice because `server.py` validates path traversal before writing files, but running as root is unnecessary and violates least-privilege.
- **Suggested fix:** Add `RUN useradd -r semgrep-user && USER semgrep-user` before the `CMD` line (assessment only — do not apply).

### Intentional Fixtures (excluded from real findings)

`fixtures/vuln-sample/app.py` contains deliberate CWE-78 (OS command injection via `subprocess.call(cmd, shell=True)`) and CWE-89 (SQL injection via string concatenation). These are the planted vuln corpus used to validate the end-to-end scan pipeline. Semgrep would flag both if the directory were scanned; they are **expected and intentional**.

Similarly, `fixtures/semgrep-output.json` and `fixtures/claude-response.json` are static test stubs; they contain no live secrets.

### Manual Code Review Notes (supplemental, no Semgrep rule covers these)

**No additional security issues were found** in the manual review of `src/`, `container/server.py`, and `public/app.js`. Specific checks performed:

- **Path traversal in container/server.py:** The `run_semgrep()` function validates that every file path resolves under the `tempfile.TemporaryDirectory()` root via `os.path.realpath` comparison before writing. This is correctly implemented.
- **API key lifecycle (src/crypto/envelope.ts + src/scan-runner/runner.ts):** The BYO API key is wrapped with AES-GCM envelope encryption (DEK wrapped by KEK, both with random IVs), stored only for the duration of the scan job, and unconditionally deleted in the `finally` block of `runScan()`. This is sound.
- **SQL injection (src/db/queries.ts):** All D1 queries use parameterized `?` bindings via the D1 prepared statement API. No string interpolation into SQL.
- **XSS in public/app.js:** All DOM mutations use `document.createElement` + `.textContent` (not `innerHTML` for user-supplied data). The one `.innerHTML = ""` call on `$("findings")` is an empty-string clear followed by `replaceChildren()`. No direct HTML interpolation of server data.
- **Input validation (src/orchestrator/validate.ts):** File count is capped at 50 and total payload at 256 KiB via Zod schema + explicit size checks. Language field is `z.string().min(1)` with no allowlist — not a security issue in the current threat model but worth noting.
- **Scan ID access control:** `/api/scans/:id` and `/api/scans/:id/sarif` have no authentication. Scan IDs are UUIDs (128-bit random), providing security through obscurity. This is appropriate for a BYO-key demo tool but is worth documenting.

## 4. Lint / Type-Check

### TypeScript (`tsc --noEmit`)

**Status: 2 errors in `src/`, 8 errors in `tests/`** (total 10 errors, exit code 2)

#### src/ errors (2) — warrant attention

| Location | Error | Analysis |
|---|---|---|
| `src/crypto/envelope.ts:21,23` | TS2345 — `crypto.subtle.generateKey()` returns `CryptoKey \| CryptoKeyPair`; `wrapKey` expects `CryptoKey` | The AES-GCM symmetric key path always returns a `CryptoKey` (not a pair), but the type system does not narrow this. At runtime the code is correct; the type error is a false negative in the type declarations. A `as CryptoKey` cast (with a comment) would suppress it without masking real bugs. |
| `src/index.ts:32` | TS2339 — `DurableObjectStub<undefined>` has no `runScan` property | The Durable Object stub is typed as `DurableObjectStub<undefined>` because the namespace binding type is not parameterized with `ScanRunner`. This is a real type gap: if `runScan`'s signature changes, the call site won't be caught at compile time. Fix: type the `SCAN_RUNNER` binding as `DurableObjectNamespace<ScanRunner>` in the `Env` interface. |

#### tests/ errors (8) — expected / environment-specific

| Pattern | Count | Analysis |
|---|---|---|
| `Cannot find module 'cloudflare:test'` | 4 | The `cloudflare:test` module is only available inside the vitest Cloudflare Workers pool (via `@cloudflare/vitest-pool-workers`). The plain `tsc` invocation doesn't know about it. These are false errors that don't affect the actual test run. The fix is to add a separate `tsconfig.test.json` with the appropriate `types` override, or exclude `tests/` from the main `tsconfig.json`. |
| `Unused '@ts-expect-error' directive` | 3 | Stale suppressions — the underlying error they were suppressing no longer exists. These should be cleaned up. |
| `Object is possibly 'undefined'` | 3 | In `tests/db/queries.test.ts` — `noUncheckedIndexedAccess` firing on array index access in test assertions. Real but minor: add `!` non-null assertions or use `.at(0)` with appropriate guards. |

### ESLint

No ESLint configuration (`eslint.config.*`, `.eslintrc.*`) exists in the project.

**Recommendation (assessment only — not implemented):** Add `eslint-config-typescript-recommended` with `@typescript-eslint/no-explicit-any` and `@typescript-eslint/no-unsafe-*` rules. The `src/db/queries.ts` file has `(r: any)` in the `getFindings` mapper and `(this.ctx as any).container` in `runner.ts` — these bypass type safety and would be flagged by stricter ESLint rules. Currently harmless but a linter would catch accidental widening if these patterns spread.
